# Incident Report — May 8, 2026

**Severity:** P1  
**Duration:** ~45 minutes  
**Impact:** All 10 members unable to log workouts via website and Siri Shortcut  

---

## Summary

After migrating from the `may-challenge` Vercel project to the new `fitness-challenge` project, the `/api/log` endpoint began returning `FUNCTION_INVOCATION_FAILED` for all requests. No workouts could be logged for the duration of the incident.

---

## Timeline

| Time | Event |
|---|---|
| ~17:00 | Migration from `may-challenge` to `fitness-challenge` Vercel project completed |
| ~17:15 | First user reports unable to log workout via Siri Shortcut |
| ~17:20 | P1 raised — confirmed website and shortcut both failing with `Missing Env Variable` |
| ~17:35 | Root cause identified — two separate issues |
| ~17:45 | Fix deployed, service restored |

---

## Root Cause

Two issues compounded simultaneously:

**1. Stale `@supabase/supabase-js` SDK in `package.json`**
The API was rewritten months ago to use native `fetch` instead of the Supabase SDK, but `package.json` still listed `@supabase/supabase-js` as a dependency. On the new Vercel project, the SDK installed and tried to initialize at function startup — before our own validation code ran — crashing the function immediately with `supabaseUrl is required`.

**2. Environment variables not verified after project migration**
When the `fitness-challenge` Vercel project was created, `SUPABASE_URL` and `SUPABASE_ANON_KEY` were not confirmed to be correctly set. The variables existed in the old `may-challenge` project but were not properly validated in the new one before going live.

---

## Fix

1. Removed `@supabase/supabase-js` from `package.json` — function now uses native `fetch` only
2. Verified and re-added `SUPABASE_URL` and `SUPABASE_ANON_KEY` to the new Vercel project
3. Force redeployed to production

---

## What We're Doing to Prevent Recurrence

- **E2E tests on every PR** — GitHub Actions now runs the full test suite against production on every pull request. A broken `/api/log` endpoint would have been caught immediately.
- **Migration checklist** — any future project migration must include verifying env vars and running a curl test before cutting over the domain.
- **Cleaner `package.json`** — removed unused dependencies so there are no hidden runtime surprises.

---

## Lessons Learned

- Never cut over a production domain without a smoke test first
- Unused dependencies in `package.json` can cause silent failures that are hard to debug
- The E2E tests we now have would have caught this before it reached users
